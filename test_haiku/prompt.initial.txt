### Type definitions – exact wording

- **finding**: a documented cause‑and‑effect observation that links a problem to a resolution, with a quantifiable outcome; OR an explicit gap, missing implementation, or unresolved issue that blocks or risks degradation.
- **pattern**: a repeatable rule derived from repeated failure that provides an automatic sequence to apply; must be consistently applied across similar scenarios.
- **rule**: a formal, consistently enforced policy with verification, enforced across all agents; violations detected in code review or automated checks.
- **milestone**: a system achievement that resolves a blocker and unblocks subsequent work, verified by measurable result.
- **capability**: a new agent function added to the ecosystem that enables a task previously impossible.
- **integration**: a cross‑agent pipeline or handshake added or altered that creates measurable workflow impact.

## Complex Procedural Section Extraction Strategy

When extracting from procedural sections (workflows, multi-step implementations, sequential phases):

1. **Identify milestone boundaries first** – Look for completion markers: "Phase X complete", "fully closed", "live", "shipped", "unblocked", "resolved"
2. **Extract the specific constraint or failure mode** – Ask: What breaks if this step is skipped? What ordering dependency exists?
3. **Capture intermediate handoffs** – Note where responsibility transfers between agents (e.g., "Yosef research → Watsan implementation → Mycsan deployment")
4. **Name patterns with action verbs** – Use verb-noun pairs: "pre-work conflict check", "background watcher auto-registration", "filter then ingest pipeline"
5. **Flag automation opportunities** – Look for "auto-", "automatic", "self-", "eliminated manual" — these signal high-value capabilities
6. **Extract gaps and unfinished work** – "not yet in production", "gap identified", "untested" are documented findings, not omissions
7. **Quantify where possible** – Include counts, dimensions, timelines: "4,965 rows", "31 channels", "25× faster", "768-d vectors"
8. **Identify security-critical protocols** – When text describes multi-step verification, enforcement, or access control sequences, extract these as **protocols** if they are enforced repeatedly (NOT one-time fixes)

## Sparse Content & Minimal Section Extraction Strategy

**When a section contains minimal text (2–5 sentences), extract exactly the same way as full sections:**
- Extract whatever concept is present, even if incomplete; partial content yields valid extraction
- Look for hidden structure in short lines: agent mentions, tool names, system calls, conditional statements
- Single-sentence decision/finding/capability is valid if it contains measurable change or new behavior
- Apply the same type definitions; do not lower the bar for sparse sections
- **Sparse sections often describe high-impact foundational work; absence of prose does NOT indicate low value**

**Critical sparse extraction rules:**
1. **Always extract from thin content.** Look for: verb phrases, agent names, tool/system names, and completion signals ("complete", "live", "added", "integrated")
2. **Extract agent actions as capabilities or decisions.** When a sparse section lists only agent actions or tool mentions (e.g., "Watsan created the envoak skill"), extract as **capability**.
3. **Extract one-line enforcement as rules.** "Secrets must be injected via env vars" → **rule** "secrets via environment variables"
4. **Extract completion signals as milestones.** "Ecosystem dedup plan executed" → **milestone** "ecosystem deduplication plan executed"
5. **Do NOT skip sparse sections.** Missing extractions from thin content directly correlate with low scores.
